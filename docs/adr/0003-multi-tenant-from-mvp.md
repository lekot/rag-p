# ADR 0003 — Multi-tenancy с первого дня через organization_id

**Статус:** Принято  
**Дата:** 2026-04-27

---

## Контекст

Продукт целится в B2B: несколько организаций на одном инстансе (SaaS/PaaS). Пользователь явно сформулировал: "если всё равно писать — зачем дважды". Добавление multi-tenancy post-factum требует болезненной миграции схемы и ретроспективного заполнения `organization_id` на всех таблицах.

---

## Рассмотренные варианты

### 1. Schema-per-tenant

Каждый тенант получает отдельную PostgreSQL-схему. Полная изоляция на уровне СУБД. Но: каждая миграция применяется к N схемам одновременно, сложность операций растёт линейно с числом тенантов. Для Alembic требуется дополнительная логика.

**Отклонено** — тяжело мигрировать.

### 2. Database-per-tenant

Максимальная изоляция. Каждый тенант — отдельный Postgres instance или база. Стоимость инфраструктуры неприемлема для bootstrap-режима: при 10 тенантах — 10 PVC, 10 connection pools, 10 операционных единиц.

**Отклонено** — стоимость инфры.

### 3. Shared database, shared schema, organization_id (выбрано)

Все таблицы с пользовательскими данными содержат колонку `organization_id UUID NOT NULL`. Каждый запрос фильтруется по `organization_id` на уровне ORM-репозитория. Permify обеспечивает авторизацию (ADR 0002). Единая кодовая база — единая схема — единый Helm chart.

---

## Решение

Принять вариант 3. Таблицы с tenant-данными: `documents`, `chunks`, `pipelines`, `pipeline_versions`, `datasets`, `dataset_items`, `runs`, `run_results`.

Правила:
- Фильтрация `WHERE organization_id = $current_org` — всегда на уровне репозитория, не в эндпоинте.
- ORM-модели содержат `organization_id` как NOT NULL FK на таблицу `organizations`.
- Integration-тест tenant isolation обязателен: negative test — tenant A запрашивает ресурс tenant B и получает 404 (не 403, чтобы не раскрывать существование ресурса).

---

## Последствия

**Плюсы:**
- Единая кодовая база для self-host, SaaS, PaaS.
- Миграции применяются один раз.
- Простой Helm chart без параметризации per-tenant.

**Минусы:**
- pgvector при большом объёме данных нескольких тенантов на одном индексе может деградировать по производительности. Это известный риск v1 (см. roadmap "Multi-tenant perf на pgvector +5 дней").

**Риск:** если один тенант генерирует аномально большой объём данных (>10M векторов), это влияет на latency для всех остальных на том же инстансе. Митигация v1 — мониторинг + квоты на количество документов per organization. Партиционирование по `organization_id` рассматривается в v2.

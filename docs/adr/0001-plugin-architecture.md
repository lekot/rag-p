# ADR 0001 — Plugin-архитектура через entrypoints и params_schema

**Статус:** Принято  
**Дата:** 2026-04-27

---

## Контекст

RAG-Platform позиционируется как лаборатория стратегий: продукту одновременно нужны несколько chunker'ов, embedding-моделей, reranker'ов и LLM-генераторов. Количество комбинаций растёт, и каждая новая реализация не должна требовать изменений в ядре.

Дополнительное требование: UI генерирует форму конфигурации плагина динамически, без хардкода полей во фронтенде.

---

## Рассмотренные варианты

### 1. Hardcoded if/else в pipeline runner

Каждый новый тип чанкера — ветка `if chunker_type == "recursive": ...`. Просто в реализации, но не масштабируется: добавление плагина требует правок ядра, тесты усложняются, open-source контрибьюторы не могут добавить свою реализацию без PR в ядро.

**Отклонено.**

### 2. LangChain-style наследование без schema

Абстрактный класс, реализации регистрируются в реестре вручную. Упрощает добавление плагинов, но нет стандартного механизма валидации параметров и нет источника правды для динамической UI-формы.

**Отклонено** — нет валидации params.

### 3. Abstract class + JSON Schema params + Python entrypoints (выбрано)

Каждый плагин — подкласс абстрактного класса с тремя обязательными элементами:

- `params_schema: dict` — JSON Schema, описывает допустимые параметры экземпляра. Используется для валидации при создании pipeline и для генерации формы в UI.
- `cost_estimate(input: PluginInput) -> CostEstimate` — оценка стоимости запуска на переданных данных (токены, деньги, latency).
- `health_check() -> HealthStatus` — проверка доступности зависимостей (API-ключи, GPU, сетевые эндпоинты).

Регистрация — через `[project.entry-points."rag_p.plugins"]` в `pyproject.toml` плагина. Ядро обнаруживает плагины через `importlib.metadata.entry_points` при старте.

---

## Решение

Принять вариант 3. Базовые классы `Chunker`, `Embedder`, `Retriever`, `Reranker`, `Generator` живут в `apps/api/src/plugins/base.py`. Реестр инициализируется один раз при старте приложения и кэшируется.

Первые встроенные реализации (`RecursiveCharacterChunker`, `OpenAIEmbedder`, `PgvectorRetriever`) регистрируются через entrypoints в `apps/api/pyproject.toml`. Сторонние плагины устанавливаются как обычные Python-пакеты.

---

## Последствия

**Плюсы:**
- Расширяемость без изменений ядра.
- `params_schema` — единственный источник правды для UI-формы и для валидации.
- Сторонние контрибьюторы публикуют плагины как отдельные пакеты.

**Минусы:**
- Каждый плагин обязан реализовать `cost_estimate` и `health_check` — overhead для простых обёрток.
- Entrypoints требуют установки пакета (`uv pip install -e .`); горячая перезагрузка в dev усложняется.

**Риск:** при большом числе плагинов обнаружение через `entry_points` при старте может замедлить cold start. Решение — ленивая инициализация или кэш реестра в Redis.

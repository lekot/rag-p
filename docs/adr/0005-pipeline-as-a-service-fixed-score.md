# ADR 0005 — Pipeline-as-a-Service: модель эксперимента с фиксированным composite score

**Статус:** Принято  
**Дата:** 2026-04-27

---

## Контекст

Ключевой product claim из opportunity.md: аналитик без кода сравнивает стратегии и получает **fixed score каждой опции**. Это основная рыночная дифференциация от конкурентов. Нужна формальная модель: что такое "эксперимент", как считается score, что показывает UI.

---

## Рассмотренные варианты

### 1. Один "best" pipeline, как у Vectara

Платформа сама выбирает лучший pipeline и скрывает детали. Проще в реализации. Но: это именно та рыночная дыра, которую мы закрываем. Аналитик не может доверять "чёрному ящику" и не может объяснить выбор бизнесу.

**Отклонено** — это и есть пробел, который мы заполняем.

### 2. Ручное сравнение run'ов пользователем

Пользователь сам запускает pipeline A, потом pipeline B, смотрит метрики вручную. Не масштабируется при большом числе комбинаций, нет автоматизации, нет leaderboard.

**Отклонено** — не автоматизировано, не масштабируется.

### 3. Experiment = декартово произведение конфигов + composite score (выбрано)

**Experiment** — это набор pipeline-конфигураций и датасет. Система автоматически прогоняет каждую валидную комбинацию плагинов через датасет и считает composite score.

**Composite score:**
```
score = (faithfulness^w1) * (answer_relevance^w2) * (context_precision^w3) * (context_recall^w4)
```
Веса `w1..w4` конфигурируемые per-experiment (по умолчанию равные). Метрики считаются через RAGAS.

**UI:** leaderboard с сортировкой по score, возможность drill-down в любую комбинацию, side-by-side compare двух run'ов.

---

## Решение

Принять вариант 3. Таблицы: `experiments`, `experiment_runs` (FK на `pipeline_version_id` + `dataset_id`), `run_metrics` (faithfulness, answer_relevance, context_precision, context_recall, composite_score, weight_config).

Combinatorial explosion митигируется:
- Бюджет эксперимента: максимальное число комбинаций задаётся пользователем.
- Приоритизация: случайный sampling или latin hypercube sampling при превышении бюджета.
- Асинхронное выполнение: все run'ы в очереди (Hatchet/Celery), прогресс в реальном времени через SSE или WebSocket.

---

## Последствия

**Плюсы:**
- Сильный и проверяемый product claim: аналитик видит числа, не мнение платформы.
- Reproducible: config_snapshot сохраняется в `experiment_runs`, score всегда можно пересчитать.
- Leaderboard — естественный UX для сравнения.

**Минусы:**
- Combinatorial explosion: 3 чанкера × 3 embedder'а × 3 reranker'а × 3 LLM = 81 комбинация. Каждая комбинация × размер датасета — значительная нагрузка на LLM API и очередь.
- RAGAS требует LLM-вызовов для eval — стоимость растёт с числом экспериментов.

**Следующий ADR (рекомендуется):** стратегия приоритизации комбинаций в эксперименте (бюджеты, sampling, early stopping по confidence interval).
